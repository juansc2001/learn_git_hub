checkout :nao altera o  historico, nao entendi muito bem mas serve para apenas dar uma olhada em como estava o meu sistema nas versoes anteriores
    git checkout main (para voltar pro comit original)




revert : revert as mudanças que foram feitas em um  commit, e cria um commit disso ou seja, nao afeta o historico, no caso a reversao que foi feita de um commit ainda esta salva no historico de commits, apenas foi criado um commit sem quelas modificações. 
por exemplo:  

existem 3 commits
8a45058 (HEAD -> main) index3(modificadas)
b91a2ef (origin/main) index2.html criado
e5a5647 anotações modificadas

se eu dou o comando "git revert b91a2ef" , sera criado um commit sem o index2 mas ainda tera o index3 pois só sera alterado as modificações que foram feitas no commit "b91a2ef (origin/main) index2.html criado" , mas o arquivo index2 nao sera perdido ele apenas esta em um commit antigo





reset  :deleta o historico entao é muito perioso. utilizar ele sem o hard ainda mantem muitos dados do futuro, geralmente usado quando retirou muitos commits

reset (id do commit) --hard  ja esse aqui realmente vai voltar tudo como era naquele commit e apaga os commits futuros entao é super perigoso 

💣 git push --force

Ele basicamente diz:

“Não me importa o que está no GitHub, sobrescreve tudo com o que eu tenho aqui.”

Ou seja:

Ignora completamente o estado do remoto
Pode apagar commits de outras pessoas (ou até seus, se você esqueceu algo)
É cego — não verifica nada antes
🛡️ git push --force-with-lease

Esse aqui é mais inteligente. Ele diz:

“Só sobrescreve se o remoto ainda estiver exatamente como eu espero.”

Na prática:

Ele verifica se o origin/main não mudou desde o seu último pull/fetch
Se alguém (ou você mesmo em outro lugar) fez push no meio tempo → ele bloqueia



git branch mostra todas as branchs criadas

git branch 'nome' cria uma nova branch apartir da branch atual

git branch -d 'nome' deleta o branch se ja tiver tido um merge

git branch -D 'nome' deleta de forma forçada o branch
